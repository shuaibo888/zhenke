## 3. 固定业务边界

### 3.1 身份和数据归属

- 商城用户在 `shop_user`；管理员和商家后台账号在 `sys_user`。
- 商家是独立经营主体，不是商城会员等级。
- 商城用户 ID 由 `ShopAccountIdentity.requireShopUserId()` 获取。
- 商家 ID 由当前后台账号解析，不能相信前端参数。
- 用户地址、购物车、订单的详情和写操作必须带当前 `userId` SQL 条件。
- 商家商品、试用、申请、订单的详情和写操作必须带当前 `merchantId` SQL 条件。
- 不可信字段：用户 ID、商家 ID、价格、库存、订单状态、报告作者、结算金额。

### 3.2 试用、报告和首页

```text
商品上架 -> 试用招募 -> 用户申请 -> 商家审核 -> 商家发货
-> 用户确认收货 -> 用户可选发布验证报告
```

- 报告发布是可选行为。
- 一个试用申请最多生成一篇有效报告。
- 首页只聚合招募中的试用和已发布报告，不是普通商品货架。

## 5. 关键已实现代码

### 5.1 商品、试用和报告状态

```text
商品：DRAFT / ON_SALE / OFF_SALE
试用活动：DRAFT / RECRUITING / CLOSED / FINISHED
试用申请：APPLIED / APPROVED / REJECTED / SHIPPED / RECEIVED / COMPLETED / EXPIRED
订单：PENDING_PAYMENT / PAID / SHIPPED / RECEIVED / CANCELLED
```

固定分类编码为 `CATEGORY_1` 至 `CATEGORY_4`。分类名称、排序和启停来自数据库，不硬编码中文名称。

### 5.2 收货地址

关键代码：`ShopUserAddressController/Service/Mapper` 和 `ShopUserAddressMapper.xml`。

- 每个用户最多 20 个地址。
- 第一个地址自动成为默认地址。
- 删除默认地址后自动补位。
- 新增、设默认和删除通过锁定用户行串行化。

### 5.3 服务端购物车

关键代码：`ShopCartController/Service/Mapper` 和 `ShopCartMapper.xml`。

```text
GET    /shop/users/me/cart
POST   /shop/users/me/cart
PUT    /shop/users/me/cart/{cartItemId}
DELETE /shop/users/me/cart/{cartItemId}
```

- 按当前用户隔离。
- 加购和改数量重新检查商品状态、库存。
- 单个商品数量为 1 至 99。
- 购物车写操作和结算都先锁用户，再锁商品。

### 5.4 订单第一阶段

关键代码：`ShopOrderController/Service/Mapper`、`ShopOrderMapper.xml`、`ShopOrderServiceTest`。

```text
GET  /shop/orders
GET  /shop/orders/{orderId}
POST /shop/orders
POST /shop/orders/from-cart
PUT  /shop/orders/{orderId}/cancel
```

已完成：

- 请求只接收商品 ID、数量和地址 ID。
- 后端重新读取并锁定商品，不信任前端价格、库存、商家或状态。
- 同商家合单，多商家拆单。
- 条件扣库存，任一失败整笔事务回滚。
- 商品名称、封面、单价和地址写入快照。
- 取消待付款订单恢复库存并写状态日志。
- 重复取消不会重复恢复库存。
- 用户详情和取消 SQL 都带 `userId`。

## 7. 订单第二阶段实现结果

### 7.1 后端

已新增：

```text
PUT /shop/orders/{orderId}/pay
GET /shop/merchant/orders
GET /shop/merchant/orders/{orderId}
PUT /shop/merchant/orders/{orderId}/ship
PUT /shop/orders/{orderId}/received
```

状态转换：

```text
PENDING_PAYMENT -> PAID       当前用户模拟支付
PAID            -> SHIPPED    当前商家发货
SHIPPED         -> RECEIVED   当前用户确认收货
```

已落实的安全约束：

1. 更新 SQL 同时带资源归属、期望原状态和 `del_flag = '0'`。
2. 模拟支付只允许当前用户操作自己的待付款订单。
3. 商家列表先解析 `merchantId`，再 `startPage()`。
4. 商家详情和发货 SQL 必须带 `merchantId`。
5. 发货请求只接收承运商、物流单号，不接收商家 ID 或目标状态。
6. 确认收货只允许当前用户操作自己的已发货订单。
7. 每次转换写一条 `shop_order_status_log`。
8. 状态不匹配时返回明确错误，不能静默成功。
9. 退款、售后不加入当前主状态机。

最小履约字段：

```text
pay_time / carrier / tracking_no / ship_time / receive_time
```

当前代码选择扩展订单表，领域对象、Mapper、Service 和前端 DTO 已保持一致。运行和联调前必须确认实际数据库已有这些列。

### 7.2 用户端

- `shopContent.ts` 增加模拟支付和确认收货 API。
- 支付、确认收货成功后使用后端返回订单更新页面。
- `SHIPPED` 展示真实承运商和物流单号。
- 不再调用 `advanceOrderStatus()` 修改真实状态。
- 保留 loading、错误提示和重复点击保护。

### 7.3 管理/商家端

- `adminApi.ts` 增加商家订单列表、详情、发货 API。
- 订单页面移除对 `seedOrders` 的运行时依赖。
- 商家只加载自己的订单。
- 不再用 `shipOrderById()` 伪造发货。
- 本阶段禁用或标记退款未接入，不调用 `refundOrderById()` 修改真实订单。

### 7.4 已补测试

- 用户不能支付、确认收货其他用户订单。
- 订单不能重复支付、重复收货。
- 商家不能查看或发货其他商家订单。
- 非 `PAID` 订单不能发货。
- 非 `SHIPPED` 订单不能确认收货。
- 每次成功转换只写一条状态日志。
- 承运商、物流单号校验。

- 管理端不再从 `seedOrders` 初始化订单。

## 8. 回归风险

- PageHelper：必须 `解析 merchantId -> startPage() -> 查询列表`，不能先分页再查唯一账号。
- 列表响应：对未知 API 数据先用 `Array.isArray` 归一化，不能直接 `.map()`。
- Upload：图片/视频 URL 使用独立 React 状态，不能把 Upload 事件对象放入 Form 字段。
- 验证码：保留 `captchaReady/captchaLoading/captchaLoadError`。
- 验证码按钮必须写 `onClick={() => void loadCaptcha()}`，不能写 `onClick={loadCaptcha}`，否则产生 TS2322。
- User-Agent：不要重新引入 Yauaa，继续使用轻量解析。
- 归属校验不能只写 Java；Mapper SQL 必须带用户或商家条件。
- 状态转换必须使用“归属 + 期望原状态”的条件更新。

### 9.1 新工作区依赖恢复与 apply_patch 性能

为解决 Windows 工作区中 `apply_patch` 每次耗时 54 至 81 秒的问题，交接前已经安全删除以下被 Git 忽略的生成目录：

```text
delivery-admin-frontend/node_modules
delivery-admin-frontend/dist
delivery-frontend/node_modules
delivery-frontend/dist
delivery-frontend/.npm-cache
delivery-backend 各模块的 target
```

清理后项目从 131923 个文件、约 1.67 GiB 降至 1498 个文件、约 9.42 MiB，`apply_patch` 新增、修改和删除均恢复到 1.99 至 3.22 秒。Full access 本身没有解决延迟，根因是工作区内生成文件数量过多。

新工作区首次需要测试或构建时再恢复依赖：

```text
npm.cmd --prefix delivery-frontend ci
npm.cmd --prefix delivery-admin-frontend ci
mvn -f delivery-backend\pom.xml clean package
```

重新安装两套 `node_modules` 后会恢复十几万个文件，`apply_patch` 可能再次变慢。日常编码时优先保持生成目录不在工作区；需要测试时再安装，完成后按同样的忽略规则和路径边界安全清理。不要为了补丁性能长期使用 Full access。

常用命令：

```text
cd delivery-backend
mvn -pl ruoyi-shop -am test
mvn clean package

cd delivery-frontend
npm test
npm run build

cd delivery-admin-frontend
npm test
npm run build
```

高效开发顺序：

1. 先读 Service 和 Mapper XML，确定状态转换、事务和归属条件。
2. 先完成后端 DTO、Mapper、Service、Controller 和测试。
3. 后端测试通过后再扩展两个前端 service DTO。
4. 页面只接真实 service，不在页面里模拟成功。
5. 先跑受影响模块测试，再跑完整构建。
6. 每次完成一个纵向切片，不要同时重构两个大型 `index.tsx`。

Windows 上 `mvn clean` 若因 JAR 文件锁失败，先停止对应本地 Java 测试进程，不要强制清理工作区。

## 10. 下一阶段任务边界

### 10.1 订单和履约

- 商家基础发货已经完成；下一步是实时物流轨迹，而不是再次实现本地模拟发货。
- 商家只能查看自己的订单；平台管理员全局订单查询仍未实现。

- 评论必须持久化到后端，同一篇报告在首页和详情页共享数据。

### 10.3 线上与线下试用

- 试用活动增加明确类型，不通过前端文案猜测线上或线下。
- 线上保留申请、审批、物流、收货流程。
- 线下需要活动地点、参与时间、名额、签到和报告资格规则。
- 两类试用都必须由后端校验用户发布甄客验的资格。
