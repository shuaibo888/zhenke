以下规则是 Codex 在本项目中工作的最高优先级约束。除非用户针对某一次具体操作给予明确授权，否则任何任务都不得突破这些边界：

### 0.1 允许执行的工作

- 阅读和静态检查现有源码、配置、文档和 SQL。
- 编写、修改和审查前端、后端、测试代码、配置文件、文档及 SQL 脚本。
- 通过源码对照、调用链检查、类型与字段核对、状态机推演、SQL 条件审查等静态方式检查代码完整性。
- 使用不会安装依赖、不会编译代码、不会运行项目、不会连接数据库、不会生成构建产物的只读检查命令。

### 0.2 未经授权严格禁止的操作

- 禁止安装、更新或重新解析依赖，包括但不限于 `npm install`、`npm ci`、`yarn install`、`pnpm install`、Maven 或 Gradle 的依赖安装与构建命令。
- 禁止执行任何前端或后端编译、构建、打包、发布命令，包括但不限于 `npm run build`、`npm run dev`、`npm start`、`mvn compile`、`mvn package`、`mvn install`、`gradle build`。
- 禁止执行会启动前端、后端、中间件或其他服务的命令。
- 禁止执行自动化测试、集成测试、端到端测试或其他可能触发编译、启动进程、写入缓存及生成产物的验证命令。Codex 可以编写和修改测试代码，但不能擅自运行。

- 如果某项验证确实必须安装依赖、运行测试、编译、打包、启动服务或执行 SQL，Codex 必须先说明具体命令、目的、可能生成的目录或文件以及影响范围。

## 1. 项目总目标

甄客商城不是普通商品货架，而是以“真实试用、真实验证报告和可信消费决策”为核心的电商平台。

核心链路是：

```text
商家入驻并发布商品
-> 发起线上或线下试用
-> 用户申请，商家审核
-> 用户完成对应试用流程
-> 用户自愿发布甄客验
-> 其他用户阅读、讨论并进入商品购买
-> 后续按报告归因结算收益或公益资金
```

长期目标包括：

- 商家、商品、试用、甄客验、评论、购物车、订单和售后形成真实后端闭环。
- 首页以招募中的试用和已发布甄客验为主要内容，而不是普通商品列表。
- 报告必须来自真实试用资格，发布始终是用户自愿行为。
- 身份、资源归属、价格、库存和状态变化全部由后端校验，前端不能模拟成功。
- 逐步接入真实支付、真实退款、物流轨迹、报告成交归因、收益和公益账本。

## 4. 已实现的核心业务规则

### 4.1 身份和资源归属

- 商城用户存放在 `shop_user`；管理员和商家后台账号存放在 `sys_user`。
- 商家是独立经营主体，不是商城用户等级。
- 商城用户 ID 由 `ShopAccountIdentity.requireShopUserId()` 获取。
- 商家 ID 必须从当前后台账号解析，不能接收并相信前端传入的商家 ID。
- 地址、购物车、订单、评论等用户资源的查询和写入必须带当前 `userId` 条件。
- 商品、试用、申请、订单等商家资源必须带当前 `merchantId` 条件。
- 状态更新 SQL 必须同时限制资源归属、期望原状态和 `del_flag = '0'`。
- 前端传入的用户、商家、作者、价格、库存、订单状态和结算金额都不可信。

### 4.2 线上与线下试用

```text
线上：发布活动 -> 用户申请 -> 商家审核通过 -> 商家发货
-> 用户确认收货 -> 用户获得报告发布资格 -> 可选发布甄客验

线下：发布活动 -> 用户申请 -> 商家审核通过
-> 用户立即获得报告发布资格 -> 可选发布甄客验
```

- `shop_trial_campaign.trial_type` 只允许 `ONLINE / OFFLINE`。
- 同一商品可以同时存在一条线上和一条线下招募，两者名额和申请互不影响。
- 同一商品、同一类型只能存在一条仍在招募、未截止且未满员的活动。
- 活动已满、已截止或被提前终止后，不再阻止同类型下一轮活动。
- 用户申请去重维度是 `campaign_id + shop_user_id`，因此可以分别申请同商品的线上和线下活动。
- 线上申请必须提供收货信息；线下申请不要求物流信息。
- 后端必须拒绝给线下申请发货或确认收货。
- 报告资格绑定具体 `trialApplicationId`，一个申请最多发布一篇有效报告。
- 报告发布不是强制任务。

### 4.3 甄客验和评论

- 首页和商品流程展示已发布的真实甄客验。
- 从首页点击报告时，不区分查看者是否为作者，统一读取 `/shop/reports/{reportId}` 和该报告的评论列表。
- 报告作者也可以从“我的甄客验”进入同一详情页和评论区。
- 评论列表允许匿名读取；发布、回复和删除需要商城用户登录。
- 评论内容最多 500 个字符。
- 回复按两层结构展示：回复归入一级评论，同时保留实际回复对象。
- 用户只能删除自己的评论或回复。
- 删除是软删除；删除一级评论时，其全部回复一起从正常查询中消失。
- 删除确认只提示“是否删除评论？”，不要向用户解释数据库保留方式。

评论接口：

```text
GET    /shop/reports/{reportId}/comments
POST   /shop/reports/{reportId}/comments
DELETE /shop/reports/{reportId}/comments/{commentId}
```

### 4.4 购物车和订单

订单主状态：

```text
PENDING_PAYMENT -> PAID -> SHIPPED -> RECEIVED
PENDING_PAYMENT -> CANCELLED
PAID -> CANCELLED（未发货取消并完成模拟退款）
```

- 创建订单只接收商品 ID、数量和地址 ID。
- 后端重新读取并锁定商品，重新计算价格并检查库存。
- 同商家商品合并为一张订单，多商家商品拆单。
- 库存使用条件更新，任一商品扣减失败时整笔事务回滚。
- 订单保存商品和地址快照，不能依赖之后变化的商品或地址记录。
- 取消允许当前用户取消自己的待付款或已支付未发货订单，并且只能恢复一次库存。
- 已支付未发货订单取消时自动审核通过并完成模拟退款。
- 模拟支付仅允许当前用户操作自己的待付款订单。
- 发货仅允许当前商家操作自己的已支付订单。
- 确认收货仅允许当前用户操作自己的已发货订单。
- 每次履约状态转换写入订单状态日志。
- 基础物流以订单中的 `carrier / tracking_no` 为准，不伪造运输节点。

主要接口：

```text
GET    /shop/users/me/cart
POST   /shop/users/me/cart
PUT    /shop/users/me/cart/{cartItemId}
DELETE /shop/users/me/cart/{cartItemId}

GET  /shop/orders
GET  /shop/orders/{orderId}
POST /shop/orders
POST /shop/orders/from-cart
PUT  /shop/orders/{orderId}/cancel
PUT  /shop/orders/{orderId}/pay
PUT  /shop/orders/{orderId}/received

GET /shop/merchant/orders
GET /shop/merchant/orders/{orderId}
PUT /shop/merchant/orders/{orderId}/ship
PUT /shop/merchant/orders/{orderId}/refund/audit

GET /shop/admin/orders
GET /shop/admin/orders/{orderId}
```

平台管理员订单接口保持只读；管理员不能复用商家发货接口修改订单。

### 4.5 当前模拟退款流程

当前支付和退款均为模拟实现，不调用真实资金渠道。

```text
PUT /shop/orders/{orderId}/refund
```

- `PENDING_PAYMENT` 订单直接取消，不产生退款记录。
- `PAID` 且未发货的订单，取消或申请退款都会自动取消订单、恢复库存并完成模拟退款。
- `SHIPPED` 订单不能申请退款，用户必须先确认收货。
- `RECEIVED` 订单申请退款后进入 `APPLIED`，必须由所属商家审核。
- 商家可同意或拒绝；同意后进入 `APPROVED` 并完成模拟退款为 `REFUNDED`，拒绝后为 `REJECTED`。
- 商家审核接口为 `PUT /shop/merchant/orders/{orderId}/refund/audit`，请求体中的 `decision` 只允许 `APPROVED / REJECTED`，审核说明最多 500 个字符。
- 退款状态与订单主状态独立；已收货订单退款后主状态仍是 `RECEIVED`。
- 所有退款更新都限制用户或商家归属、期望订单状态、期望退款状态和 `del_flag = '0'`。
- 退款操作写入独立 `shop_order_refund_log`，管理员保持只读。
- 当前仍没有真实支付渠道退款、退货物流或平台介入。

数据库迁移：

```text
delivery-backend/sql/migrations/20260717_order_refund_application.sql
delivery-backend/sql/migrations/20260720_order_refund_workflow.sql
```

## 6. 下一阶段需求

当前用户明确提出的模拟退款流程已经完成代码实现。接下来首先需要由用户在自己的环境中手动验证迁移和以下业务路径；如发现问题，下一项代码任务是根据实际错误修复，而不是先扩大售后范围：

```text
待付款订单取消
-> 已支付未发货订单取消并模拟退款
-> 已支付未发货订单直接申请退款
-> 已发货订单申请退款被拒绝并提示先确认收货
-> 已收货订单申请退款
-> 所属商家同意或拒绝
-> 同意后进入 REFUNDED，拒绝后进入 REJECTED
-> 管理员只能查看，不能审核或修改
```

用户确认当前闭环运行符合预期后，再从以下需求中确定下一项，不应由 Codex 自行扩展产品规则。

### 6.1 售后扩展与真实退款接入

在继续编码前，需要先确认以下产品规则：

- 用户申请时是否必填退款原因、说明和图片凭证。
- 已发货与已收货订单是否需要区分“仅退款”和“退货退款”。
- 商家处理时限、同意和拒绝规则。
- 商家拒绝后用户如何申请平台介入。
- 退货地址、退货物流单号和收货确认规则。
- 退款金额是否允许部分退款，优惠和运费如何分摊。
- 真实支付渠道接入前，退款成功状态如何演示或由谁确认。

规则确定后，建议新增独立售后申请、售后操作日志和退货物流数据模型，不要继续把完整售后状态塞进 `shop_order.status`。当前 `refund_status` 只承担无退货场景下的基础模拟退款状态，不应无限扩展为完整售后单。

目标状态至少需要覆盖：

```text
已申请 -> 商家处理中 -> 商家同意/拒绝
-> 等待退货/平台介入 -> 退款处理中 -> 已退款/已关闭
```

### 6.2 报告归因、收益和公益账本

当前购买流程已经能够携带报告来源信息，但真实结算闭环未完成。后续需要：

- 将订单与来源报告、报告作者稳定绑定。
- 明确有效成交、取消、售后和退款对归因的影响。
- 明确服务费计算口径和结算时间。
- 建立不可覆盖的收益流水，而不是只保存累计数值。
- 区分用户收益与公益资金，并提供后台可审计记录。
- 将用户端“我的收益”和管理端相关展示改为真实数据。

### 6.3 评论治理

当前评论量较小时一次读取全部有效评论。数据增长后需要：

- 评论分页和回复分页。
- 用户举报。
- 管理员查询、隐藏和恢复。
- 敏感内容检测与审核记录。
- 保留现有软删除和作者归属规则。

### 6.4 线下试用扩展

当前线下试用只有申请、审核和报告资格。是否增加以下字段需要产品确认：

- 活动地点和地图信息。
- 参与日期、时段和人数场次。
- 到店签到、核销码或商家确认体验完成。
- 用户爽约和资格失效规则。

在规则确认前，不要自行给线下试用增加物流或签到状态。

### 6.5 真实支付和物流

- 真实支付需要统一下单、前端拉起支付、可信回调、主动查询、验签和幂等处理。
- 支付结果只能由可信渠道回调或主动查询驱动，不能相信前端“支付成功”。
- 真实退款必须建立在售后审核结果和支付渠道退款结果之上。
- 第三方物流服务商确定后，将真实节点写入 `shop_order_logistics_event`。
- 第三方物流事件使用 `source + source_event_id` 去重。
- 第三方服务不可用时仍保留承运商和物流单号的基础展示。

## 9. 不能破坏的边界

- 首页主体仍是试用招募和甄客验，不改回普通商品瀑布流。
- 报告发布保持自愿，不能变成完成试用的强制条件。
- 线上和线下是两条独立活动记录，不能只做前端标签区分。
- 报告资格必须绑定具体试用申请，不能退回按商品 ID 粗略判断。
- 报告作者和其他用户必须看到同一个报告详情及评论区。
- 评论不能物理删除；一级评论软删除必须覆盖其全部回复。
- 订单、试用和评论的归属校验不能只写在前端或 Java 条件中，Mapper SQL 也必须限制归属。
- 状态机更新必须校验期望原状态，防止重复提交和并发覆盖。
- 退款申请和订单履约状态保持独立；当前完成结果必须明确标为模拟退款，不能描述为真实渠道退款。
- 平台管理员全局订单保持只读，商家只能操作自己的订单。
- mock 和 seed 只可用于孤立测试或演示数据，不能重新成为真实页面运行时数据源。
- 收益、公益、物流轨迹和支付结果在真实后端闭环完成前，不能把展示数据描述为真实业务结果。

## 10. 后续完成标准

一个新需求只有满足以下条件才能在交接文档中标记为“已完成”：

1. 业务规则和状态边界已经明确。
2. 后端 Controller、Service、Mapper 和数据结构形成真实闭环。
3. 身份、资源归属、期望原状态和重复请求得到保护。
4. 用户端或管理端调用真实接口，不依赖本地状态伪造成功。
5. 数据库迁移和相关测试代码随业务代码一起更新，但 Codex 不得因此擅自执行迁移或测试。
6. 错误提示、空状态和重复操作反馈与真实后端结果一致。
7. Codex 已完成源码、调用链、数据结构和 SQL 的静态检查；需要运行验证时，由用户执行，或由用户对具体命令另行明确授权。

“代码已完成”只表示代码和 SQL 已按需求实现并完成静态审查，不表示 Codex 已经编译、打包、启动、测试或执行数据库迁移。未经用户明确授权，交接结果中必须清楚标注未进行这些运行验证。
