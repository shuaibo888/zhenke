## 2. 固定业务口径

### 2.1 账号体系

- 商城用户保存在 `shop_user`，登录用户端。
- 管理员和商家后台账号保存在若依 `sys_user`，登录管理端。
- 商家不是商城用户的会员等级，而是独立经营主体和后台角色。
- 商城会员等级与若依 RBAC 角色不能混用。

### 2.2 商家数据范围

商家端列表只查询当前登录商家自己的商品、试用和相关申请，正常页面中看不到其他商家的数据。

后端必须从当前登录账号解析 `merchantId`，不能相信前端传入的商家 ID。详情和写操作仍要校验资源归属，防止绕过列表直接请求接口。

### 2.3 线上试用和验证报告

当前真实流程：

```text
商家创建并上架商品
  -> 创建并开启试用招募
  -> 商城用户申请
  -> 商家审核
  -> 商家发货
  -> 用户确认收货
  -> 用户可以选择发布验证报告
```

- 确认收货后即可发布对应试用的验证报告。
- 发布报告是可选行为，不是强制任务。
- 不发布就没有报告，也不会进入首页内容流。
- 一个试用申请最多生成一篇有效报告。

### 2.4 首页内容

首页不是普通商品货架，只聚合：

1. 状态为招募中的线上试用。
2. 用户主动发布的验证报告。

两类内容都关联真实商品。普通上架商品不能直接全部铺到首页。

### 3.5 验证报告和首页内容流

数据模型：

- `shop_verification_report`
- `shop_verification_report_resource`

发布报告时后端检查：

- 试用申请属于当前商城用户。
- 申请状态为 `RECEIVED`。
- 该申请尚未生成报告。
- 报告正文、不足、适用人群和资源符合 DTO 校验。

发布成功后报告状态为 `PUBLISHED`，申请状态更新为 `COMPLETED`。首页按分类和内容类型聚合正在招募的试用与已发布报告。

主要接口：

```text
GET  /shop/home/feed
GET  /shop/reports/{reportId}
POST /shop/reports
GET  /shop/reports/me/list
GET  /shop/merchant/reports
```

### 3.6 收货地址

已实现商城用户自己的收货地址查询、新增、编辑、删除和设为默认。后端从当前商城用户 Token 解析 `userId`，所有详情和写操作都带用户归属条件。

- 每个用户最多保存 20 个地址。
- 第一个地址自动成为默认地址。
- 删除默认地址后，后端自动选择一个剩余地址作为默认地址。
- 新增、设默认和删除操作通过锁定当前商城用户行串行化，避免并发产生多个默认地址。

主要接口：

```text
GET    /shop/users/me/addresses
POST   /shop/users/me/addresses
PUT    /shop/users/me/addresses/{addressId}
PUT    /shop/users/me/addresses/{addressId}/default
DELETE /shop/users/me/addresses/{addressId}
```

### 3.7 服务端购物车与订单第一阶段

购物车已由用户端本地状态切换为服务端持久化，并按当前商城用户隔离。新增、修改数量和结算时，后端重新检查商品状态与库存。

订单第一阶段已实现安全下单、按商家拆单、用户订单查询和待付款取消：

- 下单请求只接收商品 ID、数量和地址 ID，价格、用户、商家、库存和状态由后端确定。
- 同一商家的多个商品进入同一订单，多商家商品拆成多个订单。
- 后端在事务内锁定用户与商品，重新读取价格、库存和商家归属，并条件扣减库存。
- 商品名称、封面、单价和收货地址写入订单快照。
- 取消待付款订单时恢复库存并记录状态日志；重复取消不会再次恢复库存。
- 列表、详情和取消接口都使用当前登录用户 ID 作为查询条件。

主要接口：

```text
GET    /shop/users/me/cart
POST   /shop/users/me/cart
PUT    /shop/users/me/cart/{cartItemId}
DELETE /shop/users/me/cart/{cartItemId}

GET  /shop/orders
GET  /shop/orders/{orderId}
POST /shop/orders
POST /shop/orders/from-cart
PUT  /shop/orders/{orderId}/cancel
```

## 6. 已修复的关键代码问题

### 6.1 PageHelper 双重 LIMIT

商家列表接口曾先调用 `startPage()`，再查询唯一商家账号，PageHelper 把 `LIMIT 100` 错加到已有 `limit 1` 的账号查询上。

当前顺序统一为：

```text
解析当前 merchantId -> startPage() -> 查询真正的列表
```

商品列表、试用列表和试用申请列表都必须保持该顺序。

### 6.2 首页数组类型崩溃

`mixedItems.map is not a function` 的根因是把未知 API 数据直接作为数组使用。当前服务层和页面层都使用 `Array.isArray` 将异常数据归一化为空数组，再构造混合内容。

### 6.3 报告图片上传类型崩溃

`reportImages.map is not a function` 的根因是 Ant Design `Upload` 事件对象写入了表单字段。当前图片 URL 和视频 URL 使用独立 React 状态，发布时再组装 `resources`，不能重新改回由表单直接托管 `Upload` 值。

### 6.4 验证码加载状态

用户端现在显式维护：

```text
captchaReady
captchaLoading
captchaLoadError
```

验证码状态未确认时禁止登录和商家申请；加载失败显示重试入口。`fetchShopCaptcha` 会拒绝“已启用但缺少图片或 UUID”的不完整响应。

后端 `GlobalExceptionHandler` 已增加 `BaseException` 处理，使验证码过期、验证码错误等异常返回明确业务消息。

### 6.5 User-Agent 解析内存占用

已从父 POM 和 `ruoyi-common` 移除 Yauaa 8.1.1。`UserAgentUtils` 改为轻量正则解析，避免登录时初始化大型 matcher 数据。打包 JAR 中应保持 `YAUAA_JAR_ENTRIES=0`。

`delivery-backend/ruoyi-admin/src/main/resources/application.yml` 已关闭 devtools restart，避免打包运行时启用开发热重启。

## 10. 开发约束

- 不要恢复已删除的 `ruoyi-ui`。
- 不要把商家建模为商城会员等级。
- 不要相信前端传入的商家 ID、用户 ID、价格、库存、报告作者或状态。
- 不要把普通商品列表改成首页主体。
- 不要把确认收货后的报告发布改成强制行为。
- 不要提交 `.deploy/`、数据库数据快照、环境变量、构建产物或依赖目录。
- 修改现有状态机前先阅读 Mapper SQL 和服务层事务逻辑。
- 新增接口时同步更新前端 DTO、错误处理和测试。
