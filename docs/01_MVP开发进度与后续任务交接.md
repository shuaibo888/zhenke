# 悫者商城 MVP 开发进度与后续任务交接

> 更新时间：2026-07-15  
> 用途：供新对话快速理解当前代码状态并继续开发。  
> 范围：仅记录产品、前端、后端、数据库和联调进度，不包含部署工作。

## 1. 项目目标

当前目标不是继续维护最初收到的两个打包静态文件，而是以 `delivery-main` 源代码为基础，将悫者商城改造成一个真正的前后端分离项目。

技术结构如下：

| 部分 | 技术与目录 | 当前定位 |
|---|---|---|
| 后端 | `delivery-backend`，RuoYi-Vue Spring Boot 3，Java 17 | 权限、认证和商城业务 API |
| 用户端 | `delivery-frontend`，React 19 + Umi 4 + Ant Design 6 | 消费者注册、登录和商城业务 |
| 管理端 | `delivery-admin-frontend`，React 19 + Umi 4 + Ant Design 6 | 管理员和后续商家工作台 |
| 数据库 | MySQL，数据库名 `delivery_platform` | 若依系统表和商城业务表 |
|缓存与会话 | Redis | 若依 Token、验证码和登录状态 |

若依原有 Vue 管理端 `ruoyi-ui` 已删除。后续只维护两个 React 前端，不应恢复或继续开发 `ruoyi-ui`。

## 2. 必须先理解的账号模型

项目中存在两套不同性质的账号，不能混为一谈。

### 2.1 后台账号

- 存储在若依原生 `sys_user`、`sys_role`、`sys_menu` 等表中。
- 用于登录 React 管理端。
- 管理员认证、验证码、Token 和菜单权限沿用若依机制。
- 当前超级管理员可以管理商城用户。
- 后续商家后台账号还没有真正落库，目前商家相关数据仍以模拟数据为主。

### 2.2 商城用户

- 存储在新建的 `shop_user` 表中。
- 用于登录 React 用户端。
- 使用若依 BCrypt 密码、TokenService 和 Redis 会话，但不写入 `sys_user`。
- 通过一个较大的用户 ID 偏移量构造登录主体，避免商城用户 ID 与若依后台用户 ID 冲突。
- 登录主体带有内部权限标记 `shop:user`，用户资料接口会据此拒绝后台账号访问。

### 2.3 等级不是后台 RBAC 角色

商城用户当前有三个会员等级：

| level_id | level_code | 名称 | 前端 role |
|---|---|---|---|
| 1 | `ZHENKE` | 甄客 | `zhenke` |
| 2 | `YANZHENKE` | 验甄客 | `yanzhenke` |
| 3 | `XINZHENKE` | 信甄客 | `xinzhenke` |

这些是商城会员等级，不是若依 `sys_role` 权限角色。

- `sys_role` 决定后台账号能访问哪些管理功能。
- `shop_member_level` 表示商城用户成长等级。
- `review_eligible`、`trial_eligible` 是独立业务资格字段，后续不能只靠等级名称硬编码判断。

### 3.2 商城用户注册与登录

已实现：

- 用户名和密码注册。
- 用户名唯一性检查。
- 密码使用若依 BCrypt 加密，数据库不保存明文。
- 新用户默认等级为 `level_id = 1`。
- 登录时检查账号、密码和停用状态。
- 登录成功更新最后登录 IP、时间并写入登录日志。
- 登录成功签发若依 Token，Token 存入 Redis。
- 退出登录复用若依 `/logout`。
- 浏览器刷新后通过 Token 恢复会话。

### 3.3 商城用户个人资料

已实现：

- 获取当前登录用户资料。
- 修改昵称和头像字段。
- 修改密码并校验原密码。
- 修改密码后刷新当前 Redis 会话。

当前头像只是字符串字段方案，尚未建设正式文件上传和对象存储。较大的 Base64 图片不适合作为最终方案。

### 3.4 管理员管理商城用户

已实现：

- 分页查询商城用户。
- 按用户名、昵称、手机号、邮箱关键字查询。
- 按状态筛选。
- 按会员等级筛选。
- 查询所有启用的会员等级。
- 修改商城用户等级。
- 启用或停用商城用户。
- 停用用户时扫描并删除该商城用户已有的 Redis 登录会话，使停用立即生效。

对应若依权限：

```text
shop:user:list
shop:user:query
shop:user:edit
shop:user:status
```

超级管理员拥有 `*:*:*`，默认可以操作上述功能。普通后台角色必须在若依角色管理中授权。

### 3.5 用户端验证码

最新源码已经加入与若依管理端一致的验证码机制：

- 用户端调用若依现有 `GET /captchaImage`。
- 登录表单展示验证码图片，可点击刷新。
- 商城登录请求提交 `code` 和 `uuid`。
- 后端读取 `sys.account.captchaEnabled`，关闭验证码配置时不会强制校验。
- 后端使用若依相同的 Redis 键 `captcha_codes:{uuid}`。
- 验证码不存在时抛出过期异常，错误时抛出验证码错误异常。
- 每次登录失败刷新验证码。
- 注册完成切回登录页时刷新验证码。
- 用户退出登录后刷新验证码。

注意：这是最后一轮新增代码，尚未重新编译和手工联调，详见第 8 节。

### 3.6 日志目录跨平台修正

`ruoyi-admin/src/main/resources/logback.xml` 原来将日志目录写死为 Linux 路径 `/home/ruoyi/logs`，导致 Windows 直接运行打包 JAR 时可能因无权限而退出。

源码已改为：

```text
${LOG_PATH:-${user.home}/ruoyi/logs}
```

即优先使用环境变量 `LOG_PATH`，没有设置时使用当前用户目录。

该修改尚未重新执行 TypeScript 构建验证。

## 7. 当前接口清单

### 7.1 匿名接口

| 方法 | 地址 | 说明 |
|---|---|---|
| GET | `/captchaImage` | 若依统一验证码 |
| POST | `/shop/auth/register` | 商城用户注册 |
| POST | `/shop/auth/login` | 商城用户登录，提交用户名、密码、code、uuid |
| POST | `/login` | 若依后台账号登录 |

### 7.2 商城用户接口

请求头：`Authorization: Bearer {token}`。

| 方法 | 地址 | 说明 |
|---|---|---|
| GET | `/shop/users/me` | 获取当前商城用户资料 |
| PUT | `/shop/users/me` | 修改昵称或头像字段 |
| PUT | `/shop/users/me/password` | 修改密码 |
| POST | `/logout` | 退出登录 |

### 7.3 后台商城用户管理接口

请求头：`Authorization: Bearer {adminToken}`。

| 方法 | 地址 | 权限 | 说明 |
|---|---|---|---|
| GET | `/shop/admin/users` | `shop:user:list` | 分页和筛选商城用户 |
| GET | `/shop/admin/users/levels` | `shop:user:query` | 查询启用等级 |
| PUT | `/shop/admin/users/{userId}/status` | `shop:user:status` | 启用或停用用户 |
| PUT | `/shop/admin/users/{userId}/level` | `shop:user:edit` | 修改用户等级 |

### P0：当前基线验证

- 后端完整编译和打包。
- 用户端测试和生产构建。
- 管理端测试和生产构建。
- 手工验证管理员验证码登录。

### P1：商家账号与入驻

当前“商家入驻”只有用户端表单，“商家管理”只有模拟数据，没有真实商家表、审核流和后台账号绑定。

建议新增：

- `shop_merchant`：商家主体、联系人、营业执照、审核状态等。
- `shop_merchant_audit_log`：审核记录。
- 商家申请、管理员审核、驳回、停用 API。
- 审核通过后绑定或创建若依 `sys_user` 后台账号。
- 为商家建立若依角色，仅允许访问自己的商品、订单、报告和试用数据。

不要把商家直接设计成 `shop_user` 的第四个会员等级。商家是经营主体和后台权限身份，商城用户等级是消费者成长体系，两者职责不同。

### P2：商品与文件

建议在商家体系确定后实现：

- 商品分类。
- 商品主表、SKU、图片、库存和上下架状态。
- 商家只能管理自己的商品。
- 管理员可以查看和审核全部商品。
- 正式文件上传接口，替代头像、营业执照、商品图片的 Base64 或纯 URL 占位方案。

### P3：购物车、地址与订单

- 收货地址表和 CRUD。
- 服务端购物车，或明确采用前端本地购物车。
- 订单、订单明细、订单状态日志。
- 下单时的价格和库存快照。
- 取消、付款、发货、收货、退款基础状态机。
- 后台订单查询和发货。

MVP 阶段可以先做模拟支付，不要在核心订单模型稳定前接真实支付平台。

### P4：验证报告与试用

- 验证报告主表、图片/视频附件、审核与发布状态。
- 购买后评价资格与试用资格的服务端校验。
- 试用招募、申请、审核、寄送、截止和逾期状态。
- 报告发布后更新用户贡献统计。

### P5：等级成长与收益

- 等级晋升规则和审计记录。
- 评价资格、试用资格的授予与撤销流程。
- 报告数、有效数等真实统计。
- 收益和利润分成暂不属于当前基础 MVP，应在真实订单和报告链路稳定后再做。

## 10. 推荐的新对话执行顺序

### 第一阶段：验证并收口当前代码

1. 停止本地正在运行的后端，重新执行 Maven 完整打包。
2. 分别执行用户端和管理端测试、生产构建。
3. 修复任何由最新验证码改动引起的编译问题。
4. 启动 MySQL、Redis、后端和两个 React 前端。
5. 完整走通管理员登录、用户注册登录、等级修改和用户停用。

### 第二阶段：建设真实商家模块

先设计商家表和审核状态，再实现入驻申请、管理员审核、若依后台账号绑定。商家身份完成后，商品数据才有稳定的归属主体。

### 第三阶段：建设真实商品模块

将用户端商品列表和管理端商品管理同时替换为真实 API，形成第一个除账号体系外的完整业务闭环。

### 第四阶段：建设订单最小闭环

完成地址、下单、订单列表、管理员/商家发货和用户确认收货。支付先用模拟成功接口，重点先保证订单状态和权限正确。

### 第五阶段：接入项目特色业务

在商品和订单稳定后再实现试用招募、验证报告、等级成长，最后再讨论收益与利润分成。

## 12. 交接结论

当前项目已经不再是纯 UI 演示：账号、密码、Token、验证码、用户资料、会员等级、管理员权限和商城用户管理均已进入真实后端与数据库。

但它还不是完整商城。真正的业务数据闭环目前只完成到“账号与用户管理”，商品、商家、订单、报告、试用等核心模块仍需逐步从模拟数据迁移到数据库和 API。

下一步最重要的不是继续扩展页面，而是先验证最新基线，然后完成真实商家主体，让商品和后续订单拥有可靠的数据归属与权限边界。
